MODO storyblok · bff · cache · modo-landing-web-ui-lib

Propuesta técnica · PR #38

Storyblok via
BFF de Next + cache.

La lib no le pega a Storyblok. Le pega a una ruta de Next en modo-landing que actúa como única boca y como capa de cache. Cada release de la lib embebe el snapshot y los consumers reciben un PR automático.

La recomendación

Mergear PR #38
y armar el BFF en modo-landing.

Boca a Storyblok 1

Una sola ruta: modo.com.ar/api/cms/default-layout. La lib y los consumers no ven Storyblok.

TTL de cache en el BFF 5 min

s-maxage=300 + SWR 24h + invalidación on-demand vía webhook.

Cascade end-to-end ~30 min

Webhook → BFF → PR snapshot → release → PR de bump en consumers.

Por qué juntas: la lib sola no rinde sin el BFF, porque sino tenemos N consumers golpeando Storyblok sin control. Y el BFF sin la lib es un endpoint solitario sin caller productivo. Las dos piezas componen el loop.

05 · Pilar 02 BFF como única boca + cache

Pilar 02 · BFF + cache

5'

La ruta de Next es el cache.

El BFF (/api/cms/default-layout) es la única boca contra Storyblok. Devuelve la respuesta cacheada 5 minutos. Cuando el editor publica, el webhook invalida la cache y la siguiente call es fresh. La lib no se entera.

s-maxage=300 SWR 24h res.revalidate()

Endpoints del BFF (Next 12 · Pages Router)

  • GET /api/cms/default-layout · auth Bearer ${CMS_BFF_TOKEN} · responde { body } cacheado.
  • POST /api/cms/revalidate · invalida cache vía res.revalidate().
  • POST /api/cms/sync · webhook de Storyblok: revalida cache + dispara repository_dispatch al repo de la lib.

Una sola API key

STORYBLOK_API_KEY queda solo en modo-landing. La lib y los demás consumers usan CMS_BFF_TOKEN (token interno, rotable, distinto). Si se filtra, no compromete Storyblok.

Pilar 03 · Bundle limpio

0 KB

El consumer no habla con nada.

El JSON vive en dist/. SSG, SSR, RSC, CSR: a todos les llega lo mismo, hidratado del bundle. Sin fetch, sin handshake, sin token de BFF en cada repo.

cmsOverride opcional DeepPartial<Config> preview / A/B opt-in

Lo que NO entra al bundle

  • storyblok-js-client queda en devDeps (solo se usa en modo directo de fallback).
  • El script de build usa fetch nativo de Node 18+ contra el BFF.
  • Tamaño de dist/index.es.js: 157 KB (sin cambios respecto a 5.6.0).

Escape hatch para preview

Consumers que quieren hidratar dinámico (preview de Storyblok, A/B desde feature flag) pasan cmsOverride y la lib hace deep-merge sobre el snapshot. Sin breaking changes.

07 · Flujo end-to-end publish a PR de consumer

Distribución

Publicás en Storyblok. Cada consumer recibe un PR.

Cinco saltos automáticos. Humano interviene solo para mergear PR de snapshot y PR de bump.

01

Storyblok publish

Editor publica una story. Webhook va al BFF, no a GitHub.

02

BFF /api/cms/sync

Invalida cache + dispara repository_dispatch a la lib.

03

cms-sync.yml

Corre pnpm sync:storyblok contra el BFF (cache fresh). Si hay diff, abre PR.

04

release.yml

Mergeado el PR, semantic-release publica vX.Y.Z.

05

Bump cascade

bump-modo-landing + bump-promos-hub-site abren PRs en consumers.

Total típico end-to-end: 15 a 40 minutos. Si el webhook falla, el TTL de 5 min en el BFF garantiza que el snapshot se atrase como máximo 5 min de la realidad.

    Owner · TL + SRE (rotación de PATs)

  4. Configurar webhook "Story Published" en Storyblok

    Apunta a https://modo.com.ar/api/cms/sync con header secret. Validar con quien administra el espacio antes de tocar.

    Owner · PM + admin de Storyblok (content ops)
