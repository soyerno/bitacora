03 · Punto de partida GEO necesita SSR

Punto de partida · necesidad GEO

Para que los buscadores IA nos vean, la web tiene que ser SSR.

Cada vez más usuarios buscan en ChatGPT, Claude, Perplexity y Gemini antes que en Google. Esos asistentes scrappean la web y citan lo que pueden parsear. Si nuestra página entrega HTML vacío y arma el contenido en cliente, los crawlers IA no ven nada útil.

Lo que el negocio necesita

Visibilidad en buscadores IA y en Google.

  • Que ChatGPT, Claude, Perplexity y Gemini puedan citar MODO con info correcta.
  • Que Google indexe contenido completo de cada ruta (no shells vacíos).
  • Que la red MODO aparezca en respuestas a queries de promos, comercios, bancos.
Lo que tenemos hoy

Render mayormente cliente en Next 12.

  • Pages Router con getServerSideProps / getStaticProps en algunas rutas.
  • Prácticamente solo /blog/[slug] entrega body SSR útil al crawler.
  • El resto de las rutas devuelven shell vacío y arman el contenido en cliente tras hydration.

04 · El bloqueo SSR vs UX en Next 12

Lo que descubrimos al intentar

Cuando forzamos SSR en Next 12, se rompe la UX humana.

Probamos mover más rutas a SSR para resolver el problema de GEO. Lo que pasa es que la primera carga humana queda peor: aparece flicker en la navbar, el sidebar y las fuentes, y el layout salta antes de estabilizarse.

El síntoma

UX humana degradada.

  • Flicker visible en navbar y sidebar al cargar la primera vez.
  • Fuentes Google Fonts cargan tarde, los títulos saltan.
  • Layout shift (CLS) que también empeora Core Web Vitals.
La causa raíz

Next 12 no tiene las piezas modernas.

  • Sin streaming SSR + Suspense (lo trae Next 13+).
  • Sin next/font self-hosted (lo trae Next 13+).
  • Sin React Server Components para render sin client weight.

Conclusión: necesitamos el motor moderno (Next 16) que tiene streaming SSR, next/font y RSC. Es lo que destraba GEO sin sacrificar la experiencia humana.

05 · Pilar 01 Framework

Pilar 01 · Framework

12 → 16

Subir Next.js a la versión actual,
en Pages Router.

Salto incremental con codemods oficiales (12 → 13 → 14 → 15 → 16). Pages Router intacto. El codemod soporta saltar directo a 16 (@next/codemod upgrade latest), pero hacerlo incremental nos deja validar build entre cada step y aislar fallos. Para un repo de este tamaño la chain es la práctica segura.

~2 días con IA 1 dev + Claude Code Pages Router intacto Rollback <10 min

Qué se desbloquea

  • Performance: AVIF en next/image, next/font self-hosted, streaming SSR.
  • DX: Turbopack default en dev y build (HMR sub-segundo).
  • SEO / GEO: HTML más completo para Google y AI crawlers, Metadata API moderna.
  • Base para Fase 2: App Router y RSC quedan como opción cuando los necesitemos.

Lo que arrastra el upgrade a 16

  • React 19.2 ya no es opcional, sube en el mismo PR.
  • Turbopack default. Custom webpack: migrar o flag --webpack.
  • next lint eliminado, codemod migra a ESLint CLI.
  • middleware.ts → proxy.ts (codemod automático).
  • Node 20.9+ (ya estamos en 22.22). AMP eliminado (no usamos).

06 · Pilar 02 UI library

Pilar 02 · UI library

0

Consumidores externos de
modo-ui-lib-web.

La lib se publica en GitHub Packages, requiere auth con PAT, tiene su propio CI, sus propios version bumps. Y la consume un solo proyecto: este.

~3 días con IA −1 dep privada −1 repo a mantener

El plan

  • Mover los componentes a src/components/ui/ en modo-landing.
  • Eliminar la dependencia @playsistemico/modo-ui-lib-web.
  • Retirar GitHub Packages auth del CI y del .npmrc.
  • Archivar el repo modo-ui-lib-web con README de migración.

Costo que eliminamos

  • Doble PR cycle cuando se toca un componente compartido (lib + landing).
  • Version pinning y bumps recurrentes.
  • Pipeline CI extra (build + publish).
  • Token de GitHub Packages para cada dev y para CI/CD.

07 · Pilar 03 Styling

Pilar 03 · Styling

4 → 1

Sistemas de styling.

Hoy conviven styled-components, @emotion, CSS modules y Tailwind. Tailwind ya es el primario. Deprecamos los otros tres.

~1.5 a 2 semanas con IA migración por componente skill styled-components-to-tailwind

El plan

  • Inventario de componentes con CSS-in-JS o CSS modules.
  • Migración por feature, no big-bang. Cada PR es atómico.
  • Usar skill styled-components-to-tailwind para automatizar el 80%.
  • Quitar styled-components y @emotion/* del package.json al final.
  • Borrar archivos *.module.css conforme se migran.

Qué ganamos

  • Bundle más chico (3 runtimes menos).
  • 1 forma de stylar para nuevos devs.
  • Base lista para Tailwind 4 cuando llegue (Fase 4).
  • Tokens del design system ya están en tailwind.config.js.

13 · Harness Cómo aseguramos el upgrade

Cómo aseguramos el desarrollo

El upgrade pasa por el harness que ya usamos en cada PR.

No es proceso nuevo: workflow de 6 pasos + 8 guardianes de CI que vienen activos en modo-landing. Si algo de la migración 12→16 rompe, lo agarra antes de llegar a main.

El workflow · 6 pasos por PR

  • SDD — spec primero (qué + porqué), antes de tocar código.
  • TDD — test que falla (rojo) → código mínimo (verde) → refactor.
  • Tests en 3 niveles — unit, integración y E2E con Playwright.
  • Validación E2E — un bot recorre las pantallas antes de deploy.
  • Docs vivas — PRD, RFC, ADR en el mismo repo (no en wiki muerta).
  • Regresión automática — cada PR corre los tests de todo lo viejo.

El pipeline · 8 guardianes en CI

  • SonarCloud Quality Gate — 0 new smells, 0 hotspots, ≤3% duplication.
  • ESLint + commitlint + PR size — convención forzada por hook.
  • Dependabot + security audit — vulnerabilidades flageadas en PR.
  • Safe-chain — validator de dependencias supply-chain.
  • Playwright E2E — smoke desktop + mobile en cada PR.
  • Screenshots desktop + mobile — review visual obligatoria.
