MODO Developers Portal · Mapeo de API y gaps de DX

Qué hostea developers.modo.com.ar, dónde vive la API real, y qué falta para que sea un portal de developers moderno.

Hernán De Souza · Sr AI Engineer · 2026-05-20 · ~12 min de lectura

TL;DR

  • developers.modo.com.ar no hostea API. Es un viewer auth-gated que renderea el espacio Confluence "Developers". La API real vive en merchants.playdigital.com.ar.
  • El modelo de auth es username/password → JWT HS256 con TTL de 1 semana. No es OAuth2 client_credentials. No hay scopes. El onboarding pasa por la app MODO con SMS con código fijo conocido por el equipo en preprod.
  • Hay 5 endpoints documentados (auth, crear intención, obtener pago, split, JWKS). El resto (devoluciones, multi-merchant, V2) son páginas Confluence parciales o procedimientos manuales en paneles externos.
  • Los gaps de DX que faltan para mover esto a un portal moderno son 9 críticos (OpenAPI, sandbox público, SDKs versionados, self-service signup, etc.) y 7 importantes (playground, webhook tester, dashboard del integrador, etc.).

Encuadre y topología

Si entrás a developers.modo.com.ar sin sesión, no ves nada. Si te logueás, ves docs. Pero las docs no son docs nativas del portal: son el mismo contenido que merchants.modo.com.ar/docs, que a su vez renderea un espacio Confluence interno llamado Developers. Y la API que esas docs describen no vive en modo.com.ar: vive en merchants.playdigital.com.ar, otro dominio.

El SSR loader de la SPA confirma el encuadre: trae hardcoded una lista de bankIDS con los 16 bancos socios y arranca pidiéndole al usuario que se identifique como integrador de uno de ellos. La home no es un portal abierto al mundo, es una rampa selectiva.

Esto cambia el encuadre del roadmap. No estamos hablando de "evolucionar el portal developers": estamos hablando de convertir el viewer en un portal de developers real frente a una API que ya existe pero no tiene DX moderna. Son dos trabajos distintos con dueños distintos.

La API real

Después de pullear las páginas core del espacio Confluence (Autenticación, Creación de intención de pago, Obtener información de pago, Webhooks, Split de pagos, V2 Modificación de cuotas), este es el mapa de endpoints documentados:

Endpoints confirmados en docs (5 al 2026-05-20)
Verbo Path Propósito Auth
POST /merchants/middleman/token Generar accessToken (JWT HS256, TTL 1 semana) username/password en body
POST /merchants/ecommerce/payment-intention Crear intención de pago (QR + deeplink) Bearer token
POST /merchants/ecommerce/payment-intention Misma ruta + subPayments[] → Split de pago (solo DECIDIR 2.0/Tradicional) Bearer token
GET /merchants/ecommerce/payment-intention/{id}/data Obtener info de pago + payment_data + sub_payments Bearer token
GET /.well-known/jwks.json JSON Web Key Set para verificar firmas de webhooks público

Base URLs: Preprod https://merchants.preprod.playdigital.com.ar · Prod https://merchants.playdigital.com.ar. Los ejemplos cURL en las docs usan a veces merchants-api.playdigital.com.ar, que no matchea la Base URL declarada — gap menor de doc.

Headers obligatorios en todos los requests: Content-Type: application/json, Authorization: Bearer <token> (excepto /token y JWKS), y un detalle que vale señalar: User-Agent: <Merchant-Name> es mandatorio. Si falta, el server rechaza. Es no-estándar — típicamente UA es informativo — pero está documentado.

Modelo de autenticación

No es OAuth2. Es login con usuario y password contra /merchants/middleman/token que devuelve un JWT firmado HS256 con TTL de una semana. La doc recomienda explícitamente reutilizar el token entre requests, no pedir uno nuevo cada vez.

El onboarding de credenciales es atípico: el integrador se da de alta como usuario en la app MODO, y cuando llega el SMS de verificación, en preprod completa con un código de verificación fijo (conocido internamente) porque ese código no se envía. Es un hack documentado, no un mecanismo de self-service. No hay panel de credenciales para integradores externos.

Los webhooks van por carril aparte: MODO firma con JWS Flat (RFC-7515 §7.2.2), el integrador verifica contra el JWKS público en /.well-known/jwks.json. Esto sí es ortodoxo — la única recomendación de la doc que es security-grade.

Dos zonas críticas de incertidumbre: devoluciones (simples y split) y multi-merchant. Tienen páginas Confluence que existen pero no fueron leídas en esta corrida. Resolverlas es lo primero antes de cualquier roadmap.

Detalle importante: lo que las docs llaman "V2 Modificación de cuotas" no es un endpoint API. Es un procedimiento manual donde el integrador entra al admin de VTEX, edita el campo STORE ID del proveedor, y cambia el sufijo cc_code. El naming "V2" sugiere endpoint versionado pero no lo es. Esto va a confundir a cualquiera que entre nuevo.

Webhooks

El integrador expone una URL al onboardar la tienda. MODO postea ahí con un objeto que incluye signature en formato JWS Flat. La doc pide dos cosas en orden:

  1. Responder 200 instantáneo, antes de procesar nada.
  2. Procesar asincrónicamente en otro thread.

Y una optimización con dientes: descargar el JWKS una vez al iniciar la app, no en cada llamada al webhook. Es la primera vez que vi una doc oficial hacer ese llamado explícito — útil.

import { JWS, JWK } from 'node-jose';

let modoKeyStore;

async function initModoKeyStore() {
  const jwksUrl = 'https://merchants.playdigital.com.ar/.well-known/jwks.json';
  const response = await fetch(jwksUrl);
  const parsedResponse = await response.json();
  modoKeyStore = await JWK.asKeyStore(parsedResponse);
}

async function verifySignature(body) {
  const verificationResult = await JWS.createVerify(modoKeyStore).verify(
    body.signature
  );

  delete body.signature;
  return isEqual(
    body,
    JSON.parse(verificationResult.payload.toString())
  );
}

Gaps de DX

Si el roadmap es "que esto se parezca a un portal de developers moderno" — Stripe, Mercado Pago for developers, Plaid — esta es la deuda visible. Los críticos bloquean adopción autónoma; los importantes son fricción que ahuyenta a integradores sin contacto comercial directo.

Críticos · 9

  • Sin OpenAPI/Swagger descargable
  • Sin self-service signup de credenciales
  • Sin sandbox público sin login
  • Sin SDKs versionados en npm/PyPI/Maven
  • Sin changelog formal
  • Sin Postman/Insomnia collection
  • Sin status page público
  • Sin ratelimits documentados
  • Sin idempotency keys explícitas (más allá de externalIntentionId)

Importantes · 7

  • Sin playground interactivo
  • Sin webhook tester / replay tool
  • Sin dashboard del integrador dentro del portal
  • Sin guías diferenciales por banco (aunque los 16 IDs están hardcoded)
  • Inconsistencia de base URL en cURL examples
  • Página "Integración Backend" vacía
  • Naming "V2" confuso (no es endpoint, es procedimiento)

Lo que está bien · 5

  • Base URLs preprod/prod claras
  • Ejemplos cURL en cada endpoint
  • JWS firmado para webhooks (security-grade)
  • Política 200 OK inmediato + async bien explicada
  • Estados de transacción claros y tabulados

Bloqueos / qué falta verificar

  • No tengo credenciales del integrador. Sin login no puedo hacer POST /token y, desde ahí, descubrir endpoints no documentados vía respuestas de error o introspección.
  • No accedí a páginas Confluence de: Devoluciones (simples y split), Cancelaciones, Multi-Merchant, Integración Frontend, ni a los plugins (Tiendanube, VTEX, Magento, WooCommerce, Tiendastic, Drubbit, Tienda Negocio).
  • No grep'eé los repos internos de MODO buscando consumo de estas APIs desde dentro — esto serviría para confirmar endpoints internos no documentados públicamente.
  • El skill modo-tech-architect no tiene un Swagger MODO cacheado en references/; los patterns que trae son del proyecto promos/ChatGPT widget, no de merchants/checkout.
