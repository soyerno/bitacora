Publicar un frontend MODO en Kubernetes

De repo vacío a subdominio.modo.com.ar en producción. El runbook canónico reconstruido de los repos vivos: cómo la plataforma compartida de playsistemico convierte un values.yaml en un front desplegado, dónde encaja el basePath, y por qué la config nunca toca un .env.

Hernán De Souza · Sr AI Engineer · 2026-05-27 · ~11 min de lectura

TL;DR

  • Un front MODO no se deploya con kubectl a mano: se enchufa a una plataforma compartida. El repo aporta config declarativa; los workflows reusables (ci-ms/cd-ms) traen el Helm chart (v1.15.3) y hacen el resto.
  • La config runtime no vive en un .env: va cifrada con SOPS en parameters/<env>/, un workflow la empuja a AWS SSM, y el entrypoint.sh la baja al arrancar el container.
  • El subdominio sale de tres líneas de values.yaml — exposeName + dnsAliasRecord.enabled + istio.external → <sub>.modo.com.ar. Sin tocar Route53 ni abrir ticket, si el chart soporta alias DNS.
  • El basePath es la pieza que más se confunde: assetPrefix:"/promos" + un rewrite de _next permite servir el mismo front embebido bajo modo.com.ar/promos y en su subdominio propio a la vez.
  • Tres casos reales lo aterrizan: promos-hub (molde vivo, subdominio + subpath), modo-landing (apex, deuda de lib vieja) y aprendeatumodo (pre-infra: todavía Vite, sin helm/workflows/sops).

La plataforma, no el repo

El error mental al desplegar el primer front MODO es pensar que el repo "contiene el deploy". No. El repo contiene config declarativa; el deploy lo hace una plataforma compartida que vive en playsistemico/workflows y trae un Helm chart común. Tu repo no tiene el chart: tiene un helm/values.yaml de ~60 líneas y un helm/namespace.yaml de 6. Todo lo demás —cómo se buildea la imagen, cómo se aplica el chart, cómo se crea el DNS— está afuera, versionado y compartido entre todos los fronts.

Esto es bueno: significa que publicar un front nuevo es, en su mayoría, copiar el molde de uno que ya funciona (promos-hub) y cambiar tres nombres. Y es la razón por la que este runbook existe: el molde está implícito en los repos vivos, pero nadie lo había escrito de corrido.

Las cuatro piezas de la plataforma: imagen base modo-base:<ver>-node22 · config runtime por SOPS → SSM · deploy por workflows reusables @v1 + chart compartido · exposición por values.yaml (subdominio/apex/subpath).

Topología de un deploy

De izquierda a derecha: tu código entra en una imagen sobre modo-base, el container arranca con entrypoint.sh que baja la config de SSM, y el chart compartido expone el service detrás de un gateway Istio externo que resuelve el subdominio.

Subdominio y basePath

Hay dos maneras de exponerse y conviene no mezclarlas. El apex (caso modo-landing) sirve en la raíz: domainName: modo.com.ar + exposeAliases: ['www'], prefijado por env (develop.modo.com.ar, …, modo.com.ar). El subdominio dedicado (caso promos-hub) agrega tres líneas:

exposeName: "promoshub"
dnsAliasRecord:
  enabled: true
istio:
  enabled: true
  external: true

Eso produce promoshub.develop.modo.com.ar … promoshub.modo.com.ar, con el DNS y la regla del gateway creados por la plataforma. No hay que tocar Route53 a mano si el chart soporta dnsAliasRecord.

Caso borde — ALB legacy. Fronts viejos sin dnsAliasRecord necesitan una regla manual en payment-k8s-services/misc-alb.yml + un ticket a PE CloudOps (caso PE-3192). Confirmá qué versión de chart te toca antes de asumir que el alias DNS es automático.

El doble en promos-hub: subdominio y subpath

Acá está la pieza del basePath. promos-hub se expone doble: como subdominio propio (promoshub.modo.com.ar) y embebido dentro del apex en modo.com.ar/promos. Para que los assets resuelvan bajo el prefijo cuando entrás por el path:

// next.config.mjs
const nextConfig = {
  assetPrefix: "/promos",
  async rewrites() {
    return [
      { source: "/promos/_next/:path*", destination: "/_next/:path*" },
    ];
  },
};

Sin assetPrefix, entrando por /promos el navegador pide /_next/... (sin prefijo), el apex no lo tiene, y la página carga sin CSS ni JS. El rewrite devuelve /promos/_next/* desde /_next/* para cerrar el círculo. Si quisieras que todas las rutas vivan bajo el prefijo —incluso en el subdominio— usarías basePath en vez de assetPrefix; promos usa solo assetPrefix porque en su subdominio sigue viviendo en root.

Config runtime: SOPS → SSM

Ningún front MODO lee un .env en producción. La config viaja por una cadena de cuatro saltos, y entenderla evita el clásico "puse la variable y el pod no la ve".

Cada env tiene su propia KMS key declarada en .sops.yaml por path_regex: develop y qa comparten cuenta, preprod y prod tienen cuentas AWS propias. Dentro de parameters/<env>/ hay dos archivos: configurations.yaml (no sensible — URLs de BFFs, feature flags, keys públicas de Amplitude/Datadog) y secrets.yaml (tokens privados). Ambos terminan mergeados en process.env; la separación es para claridad y para saber qué rotar.

Gotcha de seguridad real (hallado escribiendo este runbook): un .npmrc con un token ghp_… literal y git-trackeado es un leak. El patrón correcto —el que usa promos— es //npm.pkg.github.com/:_authToken=${NODE_AUTH_TOKEN} con el token en variable de entorno (secret MODO_PKG_TOKEN en CI). Nunca el literal en disco.

Tickets JSM que dispara el deploy

Una parte del runbook no es código sino tickets. El JSM de MODO (play-sistemico.atlassian.net) tiene los service desks de MODO catalogados; un front nuevo dispara seis canónicos. El skill hermano modo-jsm-ticket arma el payload, muestra preview con la URL del portal, y POSTea con confirmación.

† El catálogo expone tres formularios homónimos en HD sd=1: rt=1175, rt=1403, rt=1404. Usar 1404 — es el validado live por modo-jsm-ticket (caso HD-11623). Los otros dos quedan deprecated.

El orden importa. Los tres de Fase 1 (#1 → #2 → #3) son secuenciales: sin repo no hay team, sin team no hay miembros. El #4 PEA se dispara solo si el chart no soporta dnsAliasRecord o si los KMS del env todavía no existen. El #5 GRC es gate de prod, no verificación post-deploy: si rebota, no hay promoción — abrirlo al entrar a preprod, no el día D. El #6 1Password con prod credentials solo después de prod desplegado.

Validación y SLAs. De los seis, sólo HD-1404 y CYBS-1577 están "validados live" en modo-jsm-ticket — los otros cuatro son catalog-only, así que el skill hermano hace fetch de campos fresh contra /rest/servicedeskapi/requesttype/{rt}/field antes de POSTear. SLA Cyber-Solicitudes (CYBS) = 72h hábiles desde aprobación (fuente: modo-jsm-ticket/SKILL.md, no figura en el catalog.md; aplica a #2, #3, #6 + opcionales CYBS). SLA OFFSEC y PEA no declarados en el portal — tratarlos como ≥72h hasta verificar.

Anti-patrón. "Ticket de Cyber para crear repo" no existe — CYBS sólo tiene operaciones de team/access; la creación de repos es HD-1404. Si abrís CYBS para crear repo, te rebotan o te mandan a HD.

La UI lib

El nombre del paquete importa porque hay dos generaciones y se confunden. Un front nuevo consume @playsistemico/modo-landing-web-ui-lib (v5.x, React 18/19 + Tailwind 4) desde GitHub Packages, junto con @playsistemico/core. La generación vieja —@playsistemico/modo-ui-lib-web@1.32— vive solo en modo-landing, que además vendoreó componentes en src/ui-lib/ tras absorber la lib (PR #1476). Eso es deuda específica de modo-landing: un front nuevo no vendorea, consume la lib publicada.

Instalación: .npmrc con el registry + NODE_AUTH_TOKEN en env, pnpm add @playsistemico/modo-landing-web-ui-lib @playsistemico/core, y el import de los tokens Tailwind 4 en globals.css.

El caso de cero: aprendeatumodo

aprendeatumodo es el ejemplo de "no tengo nada de la infra todavía". Hoy es un export de Lovable en Vite 5 (base: VITE_BASE_PATH), con la migración a Next 16 App Router en curso (branch feat/migrate-to-next16, ticket COENXT-308). Ya trae la UI lib nueva, pero le falta todo lo demás. Clasificado por severidad:

Críticos · 3

  • PAT leak: .npmrc con ghp_… git-trackeado. Rotar + scrubear history.
  • Sigue en Vite: la infra asume next start. Terminar COENXT-308 primero.
  • Sin helm/, .sops.yaml ni parameters/: no es desplegable.

Importantes · 2

  • Sin Dockerfile/entrypoint.sh/workflows: copiar de promos.
  • Lockfiles mezclados (bun + npm + pnpm): converger a pnpm.

Lo que está bien · 2

  • UI lib nueva ya integrada (modo-landing-web-ui-lib@5.5.6).
  • SDD del Next 16 ya escrito (openspec/changes/migrate-to-next16).

El orden sugerido para aprendeatumodo: terminar Next 16 → clonar la infra de promos (helm + workflows + sops) → cargar params a SSM → primer deploy a develop → smoke → promover. Es exactamente el checklist del runbook, fase por fase.

Próximo paso

El runbook quedó codificado en tres lugares para que no se pierda: el markdown source of truth en modo-landing/docs/runbook-publish-modo-frontend-k8s.md, esta R&D en la bitácora, y un skill companion triple-layer —modo-frontend-deploy (skill + agent + slash command /modo-frontend-deploy)— que ejecuta el runbook como agente: audita el repo destino, detecta gaps (incluido el PAT leak), y bootstrapea la infra copiando del molde vivo.

El primer trabajo real del agente es obvio: llevar aprendeatumodo de 1/5 a desplegado en aprendeatumodo.modo.com.ar, cuando cierre la migración Next 16. Ese run valida el runbook end-to-end y, si aparecen gaps nuevos, se destilan de vuelta al skill.
