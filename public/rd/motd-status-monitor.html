MOTD service status monitor · 12 servicios en el arranque del terminal

Cómo armé un panel de status que se renderiza al final del MOTD sin enlentecer el arranque del shell — cache-first con TTL 5min y refresh paralelo en background.

Hernán De Souza · Sr AI Engineer · 2026-05-22 · ~5 min de lectura

TL;DR

  • El MOTD del terminal ya tenía 9 themes con ASCII animado (cyberpunk, matrix, aurora, etc.) que rotan random. Le agregué un panel de status de 12 servicios al final: GitHub, Claude, OpenAI, Datadog, Cloudflare, Google Cloud, npm, Vercel + MODO Landing/Promos/Merchants/Developers.
  • Cache-first, BG refresh on stale: el shell jamás espera red. Lee ~/.cache/motd-status/status.json en ~50ms; si la cache pasó el TTL (5min), muestra los datos viejos + dispara refresh en background con (zsh -c "..." & ) &!. La próxima terminal ve datos frescos.
  • El refresh real corre 12 curls en paralelo (~800ms total, vs ~8s si fuera secuencial). 8 servicios usan el contrato statuspage.io v2 API uniforme, GCloud usa incidents.json, y los 4 MODO usan HEAD probe (2xx/3xx/4xx = up, 5xx/000 = down).
  • Tres aprendizajes durables: (a) Anthropic redirige a status.claude.com ahora; (b) promos.modo.com.ar no resuelve DNS (skip); (c) zsh imprime var=value cuando local redeclara variables ya asignadas en un loop — fix con setopt typeset_silent.

Contexto

El MOTD del terminal de mi laptop arranca con uno de 9 themes ASCII animados (cyberpunk, matrix, aurora, vaporwave, fire, ocean, steampunk, retro, default) que se elige al azar y muestra logo MODO + versiones de node/pnpm/bun + branch git + cita de una peli. Bonito, pero no funcional: ningún dato accionable.

Lo que sí miro varias veces por día es el status de GitHub (¿el push va a fallar?), Claude (¿el rate limit es real o es Anthropic caído?), npm (¿este install está raro porque mi compu o porque el registry?), y los servicios de MODO (¿el landing está respondiendo después del último deploy?). Hoy eso se traduce en abrir 4 pestañas de status pages distintas. La pregunta era simple: ¿por qué no embeber eso en el MOTD que ya se renderiza al abrir terminal?

Restricción única: el shell tiene que arrancar instantáneo. Pegarle 12 curls en serie al abrir terminal lo haría inusable. Por eso el centro del diseño no es "qué servicios" sino "cómo no bloquear nunca".

Diseño · cache-first sin bloquear

Dos archivos solos:

  • ~/.config/motd-status.sh — librería con funciones motd_status_render + motd_status_refresh + alias motd-refresh.
  • ~/.zshrc — source de la librería + llamada a motd_status_render después del theme.

La separación funcional es clave: render y refresh son dos cosas distintas.

Render (~50ms): lee ~/.cache/motd-status/status.json, formatea grid 3 columnas × 4 filas con emojis (🟢 ok · 🟡 minor · 🟠 major · 🔴 critical · ⚪ unknown), muestra footer con conteo y hint del comando refresh. Cero red, cero await. Se llama en cada apertura de terminal.

Refresh (~800ms): dispara 12 curls en paralelo (cada uno en un subshell &), espera todos con wait, agrega los resultados ordenados por nombre desde temp files zero-padded (00, 01, ... para que el glob los devuelva sortados), y escribe la cache JSON. Se llama solo cuando la cache pasó el TTL (default 5min).

El pegamento entre ambos es la regla del render: si now - cache.ts > TTL, muestra los datos viejos con badge ↻ refreshing y dispara el refresh asíncrono con un fork detachado ((zsh -c "source ... && motd_status_refresh" & ) &!). La próxima terminal lee la cache nueva. El usuario nunca espera.

Para el primer arranque sin cache, el render muestra un placeholder fetching... + dispara el refresh inicial. Cuando se abre la segunda terminal, ya hay datos.

De los 12 servicios, 11 caen en uno de tres contratos. El doceavo (Google Cloud) tiene su propia forma.

El contrato statuspage.io v2 API es ridículamente uniforme: pegarle a https://<host>/api/v2/status.json devuelve un JSON con shape {status: {indicator, description}}, donde indicator es uno de none | minor | major | critical | maintenance. Sirve para casi todo SaaS moderno. Primera cosa a probar cuando hay que monitorear cualquier servicio nuevo.

El HEAD probe para MODO funciona porque no hay status page público: curl -sI -L --max-time 4 -o /dev/null -w "%{http_code}" y se trata 2xx/3xx/4xx como UP (host responde, así sea con 404), 5xx como degraded, 000 como down. Esto es lo que evita que un merchants.playdigital.com.ar que devuelve 404 en root se marque como caído cuando en realidad está vivo y simplemente no tiene ruta raíz.

Trap encontrado: promos.modo.com.ar no resuelve DNS (curl devuelve 000). Lo descarté del set y usé modo.com.ar/promos que sí responde 301 vía istio rewrite. No todo dominio MODO existe — verificar antes de agregarlo al monitor.

Aprendizajes durables

Tres cosas que valen más que el monitor en sí mismo, porque son re-aplicables a otra cosa mañana.

1. statuspage.io es el contrato dominante

De los 8 servicios externos del monitor, 7 son statuspage.io y devuelven el mismo JSON shape. Si alguna vez tengo que armar otro monitor (un Slack bot, un widget, un dashboard interno), el primer intento siempre es: "¿este servicio tiene /api/v2/status.json?". Más útil que cualquier wrapper específico.

2. Cache-first + BG refresh es el pattern para "terminal greeting con datos"

Cualquier cosa que se renderice al arrancar el shell (system stats, status, notifs, weather) no puede esperar red. El pattern es siempre: cache en disco + TTL + render lee cache + render dispara refresh asíncrono si stale. Vale para esto, para un widget de PRs pendientes, para un contador de notificaciones de Slack, para lo que sea.

3. zsh imprime var=value cuando local redeclara en loops

Bug que me robó 20 minutos: el script imprimía key=github, name=GitHub, etc. a stdout sin un solo echo mío. Causa: local key name url kind sin asignación dentro de un loop, en la segunda iteración esas variables ya tenían valor de la primera, y zsh (sin TYPESET_SILENT) trata local foo sobre variable existente como "mostrame el valor actual". Comportamiento heredado de ksh, no pasa en bash, no pasa en zsh -f.

Fix combinable: setopt local_options typeset_silent al inicio de la función + inicializar en la declaración (local key="" name="" url="" kind="" fuera del loop, y reset dentro). Síntoma de diagnóstico rápido: si un script zsh imprime líneas name=value que no pusiste, es casi seguro esto, no set -x.

Próximo paso

Hay tres extensiones razonables, todas opcionales:

  • Más servicios MODO: agregar Central, BO portales, el SDK CDN. La regla es solo agregar lo que efectivamente miro varias veces por día; sino el panel se vuelve ruido.
  • Detalle al click: si una entrada está 🟡 o 🔴, agregar un comando motd-status <service> que abra la description completa del incident (ya está en la cache, falta solo renderizarla). Útil cuando me importa qué exactamente está roto.
  • Sound/notify: si un servicio pasa de 🟢 a 🔴 entre refreshes, mandar un terminal-notifier. Cuidadoso con el ruido — solo para los servicios críticos (Claude, GitHub).

Por ahora el panel cumple su propósito original — abrir terminal y saber en 1 segundo si lo que voy a hacer va a funcionar o si hay algo down upstream. El resto es feature creep si no aparece una necesidad concreta.
