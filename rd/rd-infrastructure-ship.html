Contexto · de RD01 a RD03

El 20 de mayo abrimos la categoría R&D en la bitácora con RD01 (mapeo del portal developers). La motivación era simple: hay un tipo de contenido que no es deck, no es RFC, no es entrada de Notion. Es research — long-form técnico con diagramas, que precede a una propuesta formal y vive público para que quede registro.

El primer artículo definió el pattern visual: hero + TL;DR + TOC sticky + prose ~70ch + diagramas SVG inline theme-aware + Chart.js leyendo CSS vars + matrix grid + gap columns + endpoint table + footer con disclaimer. Tres archivos: rd/index.html (catálogo), rd/rd.json (agent-readable), rd/research-article.css (estilos).

El problema del pattern de UN artículo: existe solo en la cabeza del que lo escribió. Si mañana lo retomo dos meses después, voy a tener que abrir rd/developers-portal-mapping.html y leer 36 KB de HTML para acordarme cómo se hace un diagrama de topología, qué clase usa el verbo POST en una tabla, o si el callout warning es callout-warn o callout-warning.

RD02 (genealogía de MODO Govern) ya empezó a forzar ese pattern. Y para que RD04, RD05, RDn no se desvíen, faltaba codificar el pattern en algo más estable que "leelo en RD01".

Qué shippeamos

Dos cambios, en dos lugares distintos:

1. Skill modo-research-article

Skill nuevo en ~/.claude/skills/modo-research-article/. Auto-invocable cuando aparecen frases como "armá un research", "RD nuevo", "mapeo de superficie", o /modo-research-article. La idea es que cualquier agente en cualquier sesión futura entre por acá, no por "fijate cómo está hecho RD01".

Estructura:

modo-research-article · archivos del skill
Path Rol
SKILL.md Workflow end-to-end (8 pasos) + 13 reglas duras.
README.md Quick start + features + anti-patterns + changelog.
assets/article-template.html Skeleton HTML con placeholders {{...}}, marker SVG global, todos los <head> wired al repo.
assets/rd-json-entry.template.json Shape canónico del item nuevo en rd.json.
references/visual-recipes.md Recetas drop-in para 8 visuales: topology, state machine, endpoint table, callouts, matrix, gap columns, Chart.js.
references/token-system.md Mapeo completo de CSS vars (light/dark) a R&D status workflow + uso en SVG y Chart.js.
references/publish-workflow.md Worktree → commit firmado → PR → sync → cleanup, con comandos copy-paste.

Regla central: el CSS canónico vive en rd/research-article.css del repo erno-modo, no en el skill. El skill referencia ese path, no duplica. Si necesitás un pattern visual nuevo, va a research-article.css en un PR aparte — nunca inline en el HTML del artículo.

2. validate_rd() en scripts/sync.py

Función nueva en el sync diario (erno-modo-sync-all, cron 18:00 ART). Read-only: chequea drift del catálogo contra el filesystem y reporta. No muta rd.json.

Tres validaciones por cada item:

  1. href existence · que rd/<href> exista en disco.
  2. status ∈ workflow · oficialmente { exploración, síntesis, decisión, archivado }. Cualquier otro valor es drift.
  3. number matchea regex · ^RD\d{2,}$. Cubre RD01, RD02, ..., RD99, RD100.

Si encuentra drift, el sync lo reporta en el log y suma una línea al commit message (- rd drift: N issue(s)). Si el cron falla en interpretar algo, lo vamos a ver en el log diario; si todo limpio, dice rd: OK.

Arquitectura · cómo se reparten responsabilidades

Hay tres piezas en juego, y cada una tiene un rol fijo:

La separación importa: si el skill autor mutara rd.json sin validación, un bug del autor entra a producción sin red. Si el validador reescribiera el JSON, perdés la fuente. Con la división actual, un autor compromete pattern; un validador compromete diagnóstico; el repo nunca se rompe de un solo lado.

Post-mortem del swarm

El plan original era usar el swarm de ruflo para ejecutar Track A (skill) y Track B (sync.py) en paralelo. Dos agents foreground con worktree donde aplicara. Lo que pasó:

Tres datos crudos:

¿Por qué falló cada uno?

  • Track A · socket disconnect: el agente arrancó bien, escribió SKILL.md completo (11k), y se cayó la conexión antes de llegar a assets/. No es un fallo de planificación — es ruido de red del transporte. Pasa.
  • Track B · rate limit: el agente recibió el prompt, hizo 15 tool calls de exploración (read sync.py, inspeccionar worktree, etc.) y antes de poder editar agotó la cuota del usuario. El prompt era largo y prescriptivo; eso aumentó tool calls preliminares.

El verdadero costo no fue el fallo en sí — fue que el reporting del swarm dijo "ambos agents terminaron" en sentido literal (terminaron de correr), no en sentido útil (terminaron el trabajo). Si yo no verificaba disco, hubiera cerrado la tarea con la mitad sin hacer.

Lecciones aplicables

No repetir · 3

  • Disparar swarm cerca del rate limit del usuario. Mirar consumo antes.
  • Prompts largos a un agent foreground para edición de archivo único — mata budget en exploración.
  • Confiar en "el agente reportó OK" sin grep al disco.

Hacer · 3

  • Para edits chicos (1 archivo, <200 líneas de cambio), edit directo es mejor que delegar.
  • Después de cualquier swarm, primer paso es ls -la + git status + git diff, no leer el reporte.
  • Si vas a delegar, achicar el contexto del prompt al mínimo accionable y dar paths absolutos.

Lo que funcionó · 3

  • Worktree por track — incluso cuando el agent crasheó, el repo principal quedó limpio.
  • Lista de TODOs explícitos antes de delegar — facilitó la recuperación post-crash.
  • Disciplina de SKILL.md + README + memoria por skill nuevo — replicable sin tener que pensar la estructura.

Próximo paso

El skill modo-research-article queda listo para dogfooding. Esta misma R&D fue su primer uso real — y sirvió para detectar dos cosas chicas que ajustar en próximas iteraciones:

  • El template del skill tiene placeholders {{...}} que hay que reemplazar a mano. Sería razonable tener un script chico que tome inputs (title, slug, date, status, etc.) y genere el HTML inicial.
  • La parte de "decidir qué visuales hacen falta" sigue siendo manual y dependiente de criterio. Una checklist tipo "¿esto justifica un research?" → "¿qué visuales usar?" → "¿qué Chart.js corresponde?" ayudaría.

Ambas son mejoras incrementales, no estructurales. La infra base está: si alguien (o algún agent) entra mañana a la categoría R&D con la pregunta "cómo se hace esto", la respuesta es /modo-research-article + leer 3 archivos cortos. No 36KB de HTML de ejemplo.

El siguiente research que vaya a salir es uno temático (no meta como este). Cuando aparezca, va a probar el skill en condiciones reales — con presión de tiempo y sin la frescura del autor que acaba de escribir las reglas.
