MODO Govern · análisis de influencias

Genealogía de un proyecto interno: qué prior art declarada e implícita lo formó, y por qué los docs cuentan dos historias que todavía no terminan de cerrar.

Hernán De Souza · Sr AI Engineer · 2026-05-21 · ~10 min de lectura

TL;DR

  • El frame de producto sale de Paperclip + OpenClaw: "si Paperclip orquesta la empresa y OpenClaw es el empleado, MODO Govern es RRHH + Compliance". Está citado literal en los docs.
  • El modelo organizacional sale de Spotify Engineering (Tribes / Squads / Chapters / Guilds con dual reporting). El de conocimiento, de Zettelkasten + Obsidian (notas atómicas, wikilinks, graph view).
  • El layer de compliance — hash chain de evidencia, policy-as-code, dashboard de empleados — hereda sin citar de sigstore/SLSA, OPA/Sentinel y Backstage/Vanta. Es la parte que más se nota en el código y menos en la doc.
  • La tensión: dos pitches conviven. Uno vende "HR + Compliance para empleados IA" (visión grande, regtech B2B). Otro vende "tool interno para auditar repos GitHub con ROI 600x" (DevSecOps interno). Misma base de código.

El objeto de estudio

MODO Govern es un proyecto interno (no público, no producto). Se describe a sí mismo como una squad automation platform: contratás empleados IA con credenciales propias, los sometés a policy-as-code y guardás toda su actividad en un log hash-chained.

Tamaño actual: 23 specs, la mayoría implementadas; algunas todavía en sprint pendiente (SPEC-112 local install CLI, SPEC-119 JumpCloud SSO foundations). El stack es React 18 + Vite + TypeScript + shadcn/ui en el front, Supabase Edge Functions (Deno) + Postgres 15 con pgvector + pgcrypto + pg_cron + pgsodium en el back. 31 edge functions, 26 migraciones, 28 tools MCP, 12 crons autónomos.

Este R&D no evalúa si el producto es bueno ni si MODO debería invertir más en él. Hace una sola cosa: leer la genealogía. Cada decisión grande del producto tiene un linaje declarado o implícito. Mapearlo sirve para entender qué decisiones se heredaron sin discutirlas y dónde el producto está tirando en direcciones distintas según qué doc leas.

Fuentes: análisis read-only sobre el directorio interno modo-govern-source-2026-04-13/ (README, CLAUDE.md, 23 specs, ARCHITECTURE, BUSINESS-CASE, PITCH-PRESENTATION, AI-SQUAD-AUTOMATION, SALES-STRATEGY). No incluye conversación con el equipo que lo construyó: es la voz que sale de los docs, no la voz que sale de las personas.

Influencias declaradas

Son las que aparecen citadas con nombre y apellido en los docs internos. No hay que adivinarlas: el README las pone, y AI-SQUAD-AUTOMATION.md arranca con la frase guía.

Paperclip + OpenClaw — la frase guía

Es la influencia que define el posicionamiento del producto. Cita literal de AI-SQUAD-AUTOMATION.md:13:

"Si Paperclip orquesta la empresa y OpenClaw es el empleado, MODO Govern es el departamento de RRHH + Compliance: contrata, entrena con el conocimiento de tus empleados reales, audita y desbloquea."

La diferenciación con Paperclip está en su propia sección de la doc (sección 6). Paperclip vende "corré tu empresa con IA"; MODO Govern vende "contratá IA que cumpla tu compliance". SPEC-106 es, literalmente, un bridge MCP para que Paperclip pueda contratar empleados Govern. Es uno de esos productos que existe por diferenciación, no en el vacío — vale subrayarlo.

Spotify Engineering Culture — modelo organizacional

SPEC-113 implementa Tribes / Squads / Chapters / Guilds con dual reporting (squad manager para qué se construye, chapter lead para cómo). Es el paper de Henrik Kniberg de 2012, traído al modelo de datos de un producto SaaS. La doc lo dice sin vueltas: "Modelar la cultura de trabajo Spotify-style que usa MODO".

Esto vale señalarlo porque es el modelo cultural real de MODO, no decoración. Govern está calzando su ontología sobre la organización que ya existe — no inventando una nueva.

Zettelkasten + Obsidian — modelo de conocimiento

SPEC-109 dice explícito: "Inspiración: Niklas Luhmann's Zettelkasten + Obsidian (markdown + [[wikilinks]] + graph view + tags)". Cada empleado humano puede construir su árbol de conocimiento personal estilo Obsidian, y ese grafo alimenta a su clon IA (SPEC-110) para que tenga el mismo conocimiento tácito.

Incluso hay import/export compatible con vault de Obsidian, y mención de un futuro Obsidian plugin: modo-govern remote. Para alguien que tiene a Luhmann en la memoria persistente de su Claude, esto se lee como afinidad real, no como buzzword.

Anthropic MCP + Claude Code — el protocolo de interop

Los 28 tools MCP y el patrón "bring your own orchestrator" salen del modelo Anthropic. ARCHITECTURE.md:19 lista como orquestadores externos válidos: Paperclip, Claude Code, Cursor, OpenClaw. La capa de agentes adentro está modelada como procesos, no como funciones — patrón Claude Code / agent SDK.

HRIS metáfora — el frame de UX

Toda la UX está framed como recursos humanos, no como agent platform: hire, onboarding wizard de 5 pasos, credential vault, per-employee budget, org chart dashboard. El vibe es Deel / Rippling / Workday — productos que ya enseñaron a un usuario corporativo a pensar "empleados con credenciales y presupuesto" como UI.

No está citado como tal en los docs (no dice "inspirado en Deel"), pero el lenguaje completo del producto es de HRIS. Lo cuento como declarada porque está en cada label del UI y en cada nombre de tabla, no porque haya un párrafo que lo declare.

Influencias implícitas

Las que no aparecen citadas pero son evidentes en el diseño técnico. Son las que más informan cómo está construido — donde están las decisiones de arquitectura que el equipo tomó "por default" porque ya las habían visto en otros productos.

sigstore / in-toto / SLSA — el hash chain de evidencia

SPEC-107 hace exactamente lo que hace la familia sigstore: log append-only con hash chain, tamper-proof, verificable. La promesa es la misma — "detecta alteraciones, exportable para compliance regulatorio (SOC 2 / PCI-DSS / SOX)". No se cita, pero el patrón es ese.

OPA / HashiCorp Sentinel / AWS Cedar — policy-as-code

SPEC-101 es policy-as-code 101: rules ejecutables que evalúan cada acción de cada agente antes de dejarla pasar. La sintaxis no es Rego ni HCL, pero la idea de gates que viven al lado del runtime sale directo de ese mundo. Las rules oficiales del producto se generan desde modo-dev-framework, mi propio framework — self-influence también cuenta.

Backstage / Cortex / OpsLevel — developer portal con org chart

El "Org Chart Dashboard" de SPEC-105, el catálogo de empleados / squads, los service-like entities con dueño y tier — todo esto es genealogía Backstage. Es la forma en que el mundo developer-portal aprendió a representar "personas + servicios + dueños" en una sola vista.

Vanta / Drata / Aikido — compliance-as-a-service

El pitch alternativo del producto (PITCH-PRESENTATION.md:36) lo vende como "auditorías automatizadas de seguridad y calidad sobre repos de GitHub con ROI 600x". Eso es lenguaje 1:1 de Vanta / Drata / Aikido: dashboard de hallazgos, tracking histórico, estandarización de configuraciones, gates automatizados. Volveremos sobre esto en la sección de tensión narrativa.

n8n / Temporal — orquestación durable

12 crons autónomos + workflows + retry policies. El stack es Supabase (no Temporal), pero el modelo mental es el mismo: workflows declarados que sobreviven a reinicios, observables, con UI. Si tuvieran que escalarlo, este es el camino más corto a Temporal.

JumpCloud sobre Okta

SPEC-119 elige JumpCloud para SSO + SAML + 2FA Push. No Okta, no Auth0. Es señal interesante de fit con MODO — JumpCloud es más barato y más liviano que Okta para una organización del tamaño de MODO, pero requiere más self-service. El equipo eligió el camino que pide más operativa propia y menos cheque mensual.

Supabase como all-in-one

El stack es Postgres + pgvector + pgcrypto + pg_cron + pgsodium Vault + Edge Functions Deno. Todo Supabase, nada AWS desnudo. Es la apuesta canónica del que arranca rápido y prefiere PaaS sobre infra propia. Tiene costo en lock-in, pero acelera meses de bootstrapping.

Lovable como gateway LLM secundario

En la lista de providers LLM aparece "Lovable AI Gateway" como uno de los multi-provider options. Es pista de que el primer prototipo nació en Lovable (la herramienta de prompting-to-app), y después se hardcodeó como gateway estable. No es decisión arquitectónica, es residuo del origen.

MODO Dev Framework — la self-influence

El producto cita explícitamente SoyErnoModo/modo-dev-framework (mi repo de meta-framework) como fuente de las rules de gobernanza oficiales. Govern aplica las reglas que yo ya había escrito antes en mi propio framework personal. Eso es self-influence, y vale anotarlo porque ata el producto a un conjunto de convenciones que no son MODO institucional — son las mías. Cuando me vaya o cambie de área, esa dependencia se va a notar.

La tensión narrativa

Esto es lo más interesante que apareció leyendo los docs uno al lado del otro. Hay dos pitches conviviendo en el mismo repo, con la misma base de código, pero con prior art distinta y objetivos distintos.

Narrativa A — "HR + Compliance para empleados IA"

Vive en AI-SQUAD-AUTOMATION.md y en el README. Lenguaje: regtech, audit trail, hash chain, persona library, knowledge graph, clonar humanos. Hereda de Paperclip + Spotify + Obsidian + sigstore. Apunta a un mercado B2B grande: vender la pieza de compliance que orquestadores como Paperclip no traen.

Narrativa B — "tool interno con ROI 600x"

Vive en PITCH-PRESENTATION.md y SALES-STRATEGY.md. Lenguaje: auditorías automatizadas, scan de repos GitHub, ahorro de horas trimestrales, setup en 15 minutos. Hereda de Vanta + Backstage + GitHub Actions. Apunta a MODO mismo como cliente: ROI medible para justificar el costo interno.

El choque: los dos pitches son razonables por separado, pero llevan al producto a lugares distintos. La A pide invertir en marketplace de personas, integraciones con orquestadores externos y certificaciones de auditoría externa. La B pide invertir en cobertura de scanners, integraciones con CI de GitHub y reporting de Slack. Las dos cosas a la vez es más equipo del que hay.

No es problema del producto, es etapa. Lo que vale señalar es que antes de cualquier movimiento de scope grande conviene elegir cuál narrativa es el norte, porque ese norte va a definir qué influencias se profundizan y cuáles se sueltan. Si gana la A, sigamos mirando a Paperclip y a sigstore. Si gana la B, soltemos eso y profundicemos Vanta.

Qué hago con esto

Este R&D queda como exploración. No propone roadmap, no pide presupuesto y no marca decisión. Solo deja el mapa servido para que cuando alguien — el dueño del producto, el TL, vos que lo estás leyendo — quiera tomar la decisión grande, no tenga que reconstruir la genealogía desde cero.

Los movimientos que sí me parece que destrabaría este material:

  • Pegarle una vuelta al posicionamiento con el equipo: ¿A o B? ¿Las dos pero con orden? ¿Una primero y la otra después?
  • Auditar dependencias del producto sobre modo-dev-framework (mi repo personal). Si Govern depende de rules que yo mantengo a título individual, eso es deuda de gobernanza propia. Vale moverlas a un repo institucional.
  • Decidir si el knowledge graph sigue siendo Obsidian-compatible o se desacopla. Hoy es valor (importás tu vault y arrancás); en escala, puede ser ancla.

Si el equipo decide alguna de estas tres, lo bajo a RFC formal. Aguardo su feedback.
